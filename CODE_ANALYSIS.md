# 코드 분석 메모 (AMI_Manager.Forms.Main.LadyBugForm)

## 1) 현재 코드의 성격
- 이 저장소의 핵심 코드는 `Program.cs` 단일 파일에 집중되어 있으며, `LadyBugForm` 안에 **UI 이벤트 처리 + 파일 I/O + 비즈니스 로직 + Excel COM 내보내기 + 이미지 렌더링**이 함께 섞여 있는 WinForms 형태입니다.
- 클래스 길이가 매우 길고(수천 라인), 메서드 수도 많아 유지보수 시 변경 영향 범위 파악이 어렵습니다.

## 2) 잘된 점
- 설정 로딩 초기화 루틴을 분리(`InitializeConfig`, `InitializeUiDefaults`, `InitializeDefectListUi`)해 부트스트랩 흐름이 비교적 명확합니다.
- `ReadValue`에서 INI 조회 캐시를 두어 반복적인 API 호출 비용을 줄이려는 의도가 보입니다.
- `SetPictureBoxImage`에서 기존 이미지를 Dispose 처리해 메모리 누수를 완화하려는 방어 코드가 있습니다.
- 패널 리스트 출력 시 `BeginUpdate/EndUpdate`와 `AddRange`를 사용해 UI 깜빡임과 성능을 고려했습니다.

## 3) 주요 리스크
- **단일 거대 클래스(신의 객체)**: 화면 제어와 도메인 처리, 파일 파싱, 수식 파서, Excel 작성이 강결합되어 테스트/리팩터링 난도가 높습니다.
- **동기식 대량 I/O**: 로그 CSV, 이미지/바이너리, Excel 내보내기가 대부분 UI 스레드 이벤트에서 직접 동작해 프리징 가능성이 있습니다.
- **하드코딩 의존성**: 컬럼 인덱스, 문자열 키(`"DefectJudge"`, `"Classify_Group"`), magic number(예: 999) 의존이 많아 스키마 변경에 취약합니다.
- **에러 처리 일관성 부족**: 일부 구간은 `MessageBox`, 일부는 `Console.WriteLine`, 일부는 무시(`catch {}`)여서 운영 중 원인 파악이 어렵습니다.
- **성능 병목 가능성**: 확대 상태에서 픽셀별 `GetPixel` + `DrawString` 반복은 고해상도 이미지에서 렌더링 비용이 큽니다.

## 4) 우선 개선 권장 순서
1. `LadyBugForm`에서 데이터 접근/파싱/도메인 로직을 서비스 클래스로 분리.
2. 검색/엑셀/이미지 로딩 루틴을 `async` + 취소 토큰으로 분리하고 UI 스레드 점유 최소화.
3. CSV/Feature 데이터를 강타입 DTO로 정리(문자열 키 딕셔너리 축소).
4. 로깅 체계 통일(파일/레벨/예외 스택 포함).
5. 렌더링 최적화(필요 시 `LockBits`, 타일링, 표시 픽셀 제한).

## 5) 빠른 점검 체크리스트
- 대량 로그 파일 검색 시 UI 응답성(검색 중 클릭/창 이동 가능 여부)
- Excel 다중 패널 추출 시 COM 객체 해제/메모리 상승 추이
- 비정상 CSV(컬럼 부족, 시간 포맷 깨짐) 입력 시 예외 처리
- 확대/이동 반복 시 `PB_MAIN` CPU 점유율
